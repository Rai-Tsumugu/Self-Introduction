1. 議題

  1. GitHub で管理する Next.js サイトを、独自ドメイン (Cloudflare DNS 管理) で公開する最適構成の確定
  2. モノレポ (self-introduction/ 配下にアプリ) への対応
  3. セキュリティ・運用上の注意点と、緊急時のロールバック設計
  4. GitHub 連携による自動デプロイ / PR プレビューの設計

2. 各エージェントの発言要旨

Agent A

DevOps / インフラ担当

セットアップ手順とインフラ的な正しさを担当

主な主張

  • モノレポ構成のため Vercel の Root Directory を self-introduction に必ず設定する。指定しないと「No Next.js version detected」で初回ビルドが失敗する。
  • 既存の Dockerfile はリポジトリにあっても Vercel では使われない (Vercel のビルダーが優先)。
  • Cloudflare 側は apex / www の両方に CNAME → cname.vercel-dns.com を DNS only (グレー雲) で登録する。apex への CNAME は Cloudflare の CNAME Flattening で許容される。
  • SSL は Vercel が Let's Encrypt から自動発行・自動更新するため、運用作業は不要。
「cname.vercel-dns.com を必ずグレー雲で登録すること。オレンジ雲にすると ACME チャレンジが遮断されて証明書発行に失敗する。これは見落としやすいので最初に潰しておきたい」

提示した検証コマンド

  • dig example.com +short で cname.vercel-dns.com へ解決されるか
  • curl -I https://example.com で server: Vercel と HSTS ヘッダ確認
  • openssl s_client で Let's Encrypt 発行・期限確認
Agent B

開発者体験 (DX) / CI 担当

GitHub 連動・PR フロー・ロールバックを担当

主な主張

  • Production Branch = main、Preview Deployments は「All branches」で有効化。feat/* 系のすべての push に Preview URL が自動発行される。
  • Vercel GitHub App がデフォルトで PR にコメント + GitHub Checks に Status を投稿するため、ブランチ保護で Vercel Checks をマージ必須化すれば壊れたコードのマージを物理的に防げる。
  • vercel.json は 今回不要。Dashboard 設定で済むものを Git で二重管理するとずれる。
  • ロールバックは Dashboard から前回安定デプロイを Promote to Production で即時復旧。その後 git revert で main を巻き戻し、コードと本番の状態を一致させる。
「Promote だけで終わらせると Git の main と本番が乖離する。必ず revert PR をセットで作って状態を合わせるのを Runbook に明記したい」
Agent C

セキュリティ / 運用担当

リスクと対策・カットオーバー設計を担当

主な主張

  • Cloudflare のプロキシ ON は SSL の二重化・ACME 失敗・実 IP 喪失 など害が大きい。CDN/WAF を Cloudflare で効かせたいなら、むしろホスティングごと Cloudflare Pages に寄せる方が筋がよい。本構成では Vercel Firewall を代替に使う。
  • 環境変数は Production / Preview / Development を必ず分離、シークレットは Sensitive フラグを立てる。NEXT_PUBLIC_ はクライアントに露出するためシークレットには絶対付けない。
  • *.vercel.app の Preview URL は推測・SNS 流出のリスクがあるため、未公開コンテンツを含むなら Vercel Authentication を Preview に強制すべき。
  • Cloudflare の DNSSEC を有効化し、レジストラに DS レコードを登録する。
  • カットオーバー前に DNS TTL を 300 秒に短縮しておき、切り戻し時間を最小化する。
「TLS 二重化と ACME 失敗はトラブルシュート時に原因が見えにくい。"Cloudflare はグレー雲固定" を運用ルールとしてドキュメントの最上段に書いておくべき」
⚠ 強い警告: Cloudflare のオレンジ雲 (Proxied) は本構成では絶対に使用しない。

3. 論点と合意

論点結論提案者
Cloudflare プロキシの ON/OFF OFF (DNS only / グレー雲) で固定。例外を作らない。 DevOps / セキュリティ
モノレポへの対応 Vercel の Root Directory に self-introduction を指定。Dockerfile は使わない。 DevOps
vercel.json の要否 今回は 不要。Dashboard 設定で完結させる。 DX
環境変数の管理方針 3 スコープ分離 + Sensitive フラグ + vercel env pull でローカル同期。 セキュリティ
マージ品質ゲート main へのブランチ保護 + Vercel Checks をマージ必須化。 DX
ロールバック手順 Promote to Production で即時復旧 → git revert PR で履歴を一致。 DX

4. 決定事項 (Decision)

Decision

本案件のデプロイ構成は以下で確定

本議事録の合意をもって正式採用
  1. ホスティング: Vercel (Hobby プラン、Root Directory = self-introduction)
  2. DNS: Cloudflare (apex / www とも CNAME → cname.vercel-dns.com / DNS only)
  3. CI: Vercel GitHub App による自動デプロイ + PR プレビュー、main ブランチ保護で Vercel Checks 必須化
  4. セキュリティ: DNSSEC 有効化、Sensitive 環境変数分離、Next.js 側でセキュリティヘッダ付与
  5. 運用: TTL 300 秒で切替 / Promote to Production でロールバック / Runbook を Docs に常備

  • CSP (Content-Security-Policy) の具体ポリシー設計 — 依存する外部リソース (フォント / アナリティクス) 確定後
  • Preview URL を社外レビューに渡す場合の認証方式 — Vercel Authentication / Password Protection
  • 本番アクセスログ / メトリクスの可視化 — Vercel Analytics 採用可否
  • 将来 API / DB を足す場合の構成 — Vercel Functions か Cloud Run 併用か
