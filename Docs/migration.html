移行手順書: 現環境 → Vercel + Cloudflare DNS

Self-Introduction プロジェクト / 人間用ビジュアル仕様書 (2026-05-21)

現在ローカルで docker compose up 起動している Next.js サイトを、 Vercel ホスティング + Cloudflare DNS (DNS only) + GitHub 連動 の本番構成に移行するための手順書です。 各ステップに 「誰が行うか」のタグを付けています。Agent (Claude Code) が実行できるものと、人間が UI で行う必要があるものを明確に区別しています。

AGENT
Agent が自動実行可能（破壊的でないコマンド・ファイル編集など）
AGENT:確認後
Agent が実行するが、内容を提示して承認後に行う（コミット・PR 作成・ファイル削除など）
MANUAL
人間が UI 操作で行う。Agent は代行不可（ブラウザログイン・ダッシュボード設定など）
VERIFY
Agent が検証コマンドを実行し結果を報告（dig / curl / openssl など）

AGENT:確認後

Phase A事前棚卸し

A-1AGENT

現状の構成確認

Agent が以下のコマンドを並列実行し、リポジトリ状態・モノレポ構造・現行 docker-compose.yml の中身・.env 系ファイルの有無を 1 メッセージで要約報告します。

git status / git branch / git branch -r / git remote -v
ls self-introduction/
cat self-introduction/package.json
cat docker-compose.yml
📌 ハマりやすい点: リモートに main ブランチが push されていないケース (git branch -r に origin/main が出ない) がある。この状態だと gh pr create --base main が Base ref must be a branch で失敗する。先に git push origin main を実施すること。
A-2AGENT:確認後

ドメイン情報のヒアリング

Agent から以下を質問します:

  • 独自ドメイン名 (例: example.com)
  • apex / www のどちらを Primary にするか
  • 既存サービスが稼働中か、許容ダウンタイム
  • Cloudflare の DNS 編集権限を持っているか

取得した値は以降の手順で ${DOMAIN} / ${PRIMARY} として一貫使用されます。

A-3AGENT

.gitignore の安全確認

.env / .env.local が ignore されているかを確認。漏れがあれば AGENT:確認後 で追記 PR を提案します。

Phase B移行前のコード整備

B-1AGENT:確認後

next.config.ts にセキュリティヘッダ追加

HSTS / X-Frame-Options / X-Content-Type-Options / Referrer-Policy / Permissions-Policy を追加。差分を提示し承認後に編集します。

B-2AGENT

ローカルでビルド検証

Vercel と同条件で通るか事前確認。失敗時はここで停止し原因を報告します。

cd self-introduction && npm ci && npm run build
B-3AGENT:確認後

作業ブランチ作成 + コミット

ブランチ名: chore/vercel-migration。Conventional Commits + 日本語本文 + git add は個別指定。

B-4AGENT:確認後

PR 作成（マージはまだ）

gh pr create で PR を作成。Vercel 連携後に Preview URL を確認してからマージします。

Phase CVercel セットアップ

Phase C はすべて Vercel Web UI でのログイン操作が必要なため、原則 MANUAL です。Agent は手順テキストを提示し、各ステップ完了を待ってから次に進みます。
C-1MANUAL

Vercel アカウント + GitHub 連携

  1. https://vercel.com/signup を開く
  2. Continue with GitHub をクリック
  3. GitHub OAuth 認可画面で Authorize Vercel
C-2MANUAL

プロジェクト作成

  1. Vercel ダッシュボード → Add New → Project
  2. Self-Introduction リポジトリの Import
  3. Root Directory を self-introduction に変更（最重要）
  4. Framework Preset が Next.js と表示されることを確認
  5. Build / Install / Output は空欄のまま
  6. Deploy をクリック
完了したら発行された *.vercel.app の URL を Agent に共有してください。
C-3MANUAL

プロジェクト設定の追記

  • Settings → General → Node.js Version: 20.x
  • Settings → Git → Production Branch: main
  • Settings → Git → Preview Deployments: All branches
  • Settings → Git → Comments on PRs / Commit Comments を ON
C-4MANUALAGENT 補助

環境変数の登録

Agent が grep -rE "process\.env\." self-introduction/ でアプリ内の参照を走査し、登録漏れ候補を提示します。実際の登録は Vercel UI で MANUAL:

  • Settings → Environment Variables で Production / Preview / Development を分離
  • シークレットには Sensitive フラグを立てる
  • NEXT_PUBLIC_ プレフィックスはシークレットには絶対付けない
現状の docker-compose.yml は NODE_ENV=production のみ。これは Vercel が自動付与するため登録不要です。
C-5MANUAL

独自ドメインを Vercel に登録

  1. Settings → Domains → Add で ${DOMAIN} を入力
  2. 続けて www.${DOMAIN} も追加
  3. Primary を選択、もう一方は Redirect に設定
  4. 「Invalid Configuration」と表示されますが、この時点では正常です（DNS 未設定のため）
Vercel が指示する CNAME ターゲット cname.vercel-dns.com を Agent に共有してください。

Phase DDNS 切替

D-1AGENT

現行 DNS をスナップショット保存

切戻し用に既存レコードを Docs/dns-snapshot-YYYYMMDD.txt に保存します。

dig ${DOMAIN} +short
dig www.${DOMAIN} +short
dig ${DOMAIN} TXT +short
dig ${DOMAIN} MX +short
D-2MANUAL

Cloudflare で TTL を 300 秒に短縮（切替前夜）

切戻しを高速化するため、既存レコードの TTL を 300 秒 (5 min) に変更しておきます。

D-3MANUAL

Cloudflare DNS レコード切替

  1. 既存の apex / www 向け A / AAAA / CNAME を削除（スナップショットは D-1 で保存済み）
  2. 以下 2 件を追加:
TypeNameTargetProxy
CNAME@cname.vercel-dns.comDNS only (グレー雲)
CNAMEwwwcname.vercel-dns.comDNS only (グレー雲)
⚠ 絶対にオレンジ雲 (Proxied) にしない。 ACME チャレンジ失敗で SSL 発行が止まり、二重 CDN による無限リダイレクト、実クライアント IP 喪失といった障害が同時多発します。
D-4VERIFY

DNS 伝播確認

dig ${DOMAIN} +short
dig www.${DOMAIN} +short
dig @1.1.1.1 ${DOMAIN} +short

期待: cname.vercel-dns.com 経由の A レコードが返ってくる。Agent が結果を要約報告します。

D-5MANUAL

Vercel 側でドメイン検証完了を確認

  • ${DOMAIN} / www.${DOMAIN} が Valid Configuration になるまで Refresh
  • SSL が Valid になることを確認（Let's Encrypt 自動発行）

Phase E切替後検証

E-1VERIFY

HTTPS / リダイレクト / ヘッダ確認

curl -I https://${DOMAIN}
curl -I https://www.${DOMAIN}
curl -I http://${DOMAIN}
E-2VERIFY

SSL 証明書確認

echo | openssl s_client -connect ${DOMAIN}:443 -servername ${DOMAIN} 2>/dev/null \
  | openssl x509 -noout -issuer -subject -dates

Issuer = Let's Encrypt、有効期限 約 90 日後 を確認。

E-3MANUAL

外部スキャン

  • securityheaders.com で A 以上
  • SSL Labs で A 以上
E-4MANUAL

主要ページ目視確認

  • トップページが正常表示
  • /_next/image 経由の画像最適化が動く
  • クライアントナビゲーションが動く
  • PR 作成で Preview URL が自動コメントされる
E-5MANUAL

GitHub ブランチ保護設定

GitHub → Settings → Branches → Add rule:

  • Pattern: main
  • Require a PR before merging
  • Require status checks → Vercel を追加
  • Require branches to be up to date
E-6MANUAL

Cloudflare DNSSEC 有効化

Cloudflare → DNS → Settings → Enable DNSSEC。表示された DS レコードをレジストラ側に登録します（レジストラ画面は Agent 代行不可）。

Phase F旧環境クリーンアップ

切替成功と最低 1 週間の安定運用を確認した後に実施。
F-1AGENT:確認後

docker-compose.yml / Dockerfile の扱い

「残す / アーカイブ / 削除」の 3 択を提示します。ローカル開発で Docker を使い続けるなら残し、それ以外は削除推奨。

F-2AGENT:確認後

README 更新

ローカル起動方法を npm run dev ベースに更新、デプロイフローと本番 URL を追記します。

F-3MANUAL

旧サーバの停止（あれば）

旧ホスティングがある場合の停止操作はサービスごとに UI が異なるため人間が実施します。

F-4AGENT:確認後

Cloudflare TTL を Auto に戻す
